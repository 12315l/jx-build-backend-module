## 1. Entity location and shape

Use this reference before generating or changing an Entity, DTO, VO, table, or database migration plan.

Place a persistent Entity at:

```text
base-framework/src/main/java/system/store/functionModule/<PascalName>Module/model/entity/<PascalName>.java
```

Use the package:

```text
system.store.functionModule.<PascalName>Module.model.entity
```

Follow inspected current-platform annotations where applicable:

- Lombok data access annotation used by surrounding modules.
- `@TableName` for the exact business table.
- `@TableId` for the primary key.
- `@TableField` when the Java property and column need an explicit mapping.
- `@TableLogic` only when the module actually uses logical deletion.
- Current schema/documentation annotations.
- Current low-code Entity marker or mapping capability when the management page requires it.

Do not copy annotations whose capabilities were disabled in the module specification.

## 2. Field selection

Generate fields from the completed module specification and inspected schema. Every persisted field must have:

- One business meaning.
- A stable Java name and database column name.
- A type and nullability decision.
- A creator of truth: client input, server assignment, calculation, or related process.
- Create and edit rules.
- A query/display need when applicable.

Do not retain `type`, `sort`, `status`, creator, or other KeyModule fields merely because the scaffold contains them. Keep only fields required by the target module or platform infrastructure.

Do not add speculative fields for future use. Record unresolved needs in the specification instead.

Associated display values such as user nickname, coach name, course title, item name, or team name normally belong in a VO or DTO query result, not the stored relation Entity.

## 3. Server-owned and client-writable fields

Classify every field:

- Client-writable on create.
- Client-writable on edit.
- Server-owned.
- Read-only calculated/display value.

Typical server-owned fields include:

- Primary key when generated by the database/platform.
- Current user, creator, operator, reviewer, or owner identity derived from the session.
- Workflow state.
- Available stock, totals, balances, final scores, and derived ratings.
- Create time, update time, completion time, and confirmation time.
- Logical-deletion marker.

Exclude protected fields from input DTOs. If an Entity is accepted directly for simple CRUD, explicitly overwrite or ignore all server-owned values before persistence.

## 4. Types, dates, quantities, and states

Choose types from the current project conventions and real data range:

- Use integer identifiers when related modules use integer identifiers.
- Use decimal types for values that require decimal precision; do not use floating-point types for money.
- Use integer quantities with positive-value validation for stock, attendance counts, capacity, and item counts.
- Use date/time types consistent with current serialization and database columns.
- Use text length appropriate to title, summary, remarks, and rich content.

Define the business meaning of every type or state value in the module specification and SQL comments. Do not expose raw numeric meanings in user-facing documentation.

A general enable/disable status may use the platform's common pattern. A workflow status must represent only confirmed transitions and must not be editable through generic CRUD.

## 5. Logical deletion and common fields

Use logical deletion only when the surrounding module, query behavior, and recovery requirements support it. If retained:

- Entity mapping and SQL column/default must agree.
- Normal queries exclude deleted rows through the platform behavior.
- Details, custom SQL, uniqueness, and recovery behavior are checked explicitly.
- Recovery endpoint is generated only when requirements enable recovery.

Do not assume a logical-delete unique record can always be recreated. A database unique constraint may still include deleted rows; decide whether the old record should be restored, rejected, or uniquely versioned.

Use creator and timestamps only when needed by platform auditing or business rules. The Service assigns server-owned values.

## 6. DTO and VO separation

Place DTOs and VOs only when needed:

```text
base-framework/src/main/java/system/store/functionModule/<PascalName>Module/model/dto/<Purpose>DTO.java
base-framework/src/main/java/system/store/functionModule/<PascalName>Module/model/vo/<Purpose>VO.java
```

Use DTO for create/edit/action input boundaries. Use VO for combined, calculated, aggregated, or display-only output.

Do not create empty DTO/VO classes. Each class name should state its purpose, such as create, edit, confirm return, match performance, page item, details, or statistics result.

Validation annotations may enforce input shape. Business ownership, state, uniqueness, stock, and cross-record rules remain in the Service.

## 7. Table design

Place module SQL at:

```text
base-framework/src/main/java/system/store/functionModule/<PascalName>Module/db/c_<snake_name>_table.sql
```

Use the actual current project table prefix and naming convention. Define:

- Table name and business comment.
- Primary key and generation strategy.
- One column for each persisted Entity property.
- Nullability and defaults matching create behavior.
- Column comments explaining business meaning.
- Required unique constraints and indexes.
- Permission/menu inserts enabled by the Controller plan.

Entity property, SQL column, Java type, SQL type, nullability, default, and state meaning must agree.

Do not add physical foreign keys automatically when the current platform does not use them. Validate relationships in the Service and add only constraints compatible with the inspected project and deployment process.

## 8. Constraints and indexes

Use database uniqueness for stable business invariants when compatible with logical deletion and lifecycle rules. Current project patterns demonstrate composite uniqueness such as:

- User plus course.
- User plus schedule.
- Team plus student.
- User plus year.

Do not copy one of these constraints into an unrelated module. Derive the exact columns from the business invariant.

Add ordinary indexes for confirmed frequent filters, joins, scopes, and ordering where they materially improve real queries. Avoid indexing every column.

Before adding a constraint or index, verify:

- Existing data will satisfy it.
- Its column order supports the actual query.
- Logical deletion does not conflict with intended reuse.
- The generated name does not collide with existing schema objects.

## 9. New tables and existing tables

Separate three facts:

1. SQL file generated.
2. SQL executed against a database.
3. Result verified through schema inspection and behavior tests.

Never report the latter two merely because a SQL file exists.

For a genuinely new development table, the current KeyModule SQL may be used as a scaffold. A destructive `DROP TABLE IF EXISTS` statement is allowed only in an explicitly approved disposable initialization context and must not be executed automatically.

For an existing table:

- Inspect its current definition and data dependencies.
- Produce an additive or migration-oriented change plan.
- Do not replace it with a drop-and-recreate script.
- Preserve unrelated columns, indexes, and user data.
- Report manual migration or backup requirements.

## 10. Alignment checklist

- Entity path, package, class, and table name align.
- Every Entity field maps to a real table column.
- Java and SQL types, nullability, defaults, and comments agree.
- Only actual module fields remain from KeyModule.
- Client DTOs exclude server-owned fields.
- Associated display data uses VO/DTO rather than fake persisted fields.
- State meanings and allowed writers are documented.
- Logical deletion behavior is consistent in Entity, SQL, Service, and custom queries.
- Unique constraints represent real invariants.
- Indexes support actual filters, joins, scopes, or ordering.
- Existing tables are changed without destructive recreation.
- Import/export annotations exist only when enabled.
- Generated, executed, and verified SQL are reported separately.
- No placeholder column, comment, or example field remains.
