# 单模块新增需求表达模板

当用户表示“想增加一个模块，但不知道如何描述”时，先帮助用户整理需求，不立即生成代码。

## 用户最少只需提供

- 模块名称：准备增加什么管理模块。
- 使用角色：哪些角色可以使用，是否仅管理员使用。
- 核心操作：希望完成哪些查看或维护操作。
- 记录内容：每条记录需要保存哪些业务信息。

如果用户已经提供项目和现有流程，主动检查可确认的信息；只有权限、业务关联、状态流程或数据结构仍会影响结果时，才集中询问一次。

## 一段式通用模板

使用 `$jx-build-backend-module`，新增“【模块中文名称】”模块，【仅供/同时供】【角色名称】使用。【角色】可以【新增、修改、删除、分页查询、查看详情或真实业务动作】，并可按【筛选条件】进行查询。每条【业务记录名称】包括【业务字段清单】。本模块【独立运行，不与……建立关联/与……建立明确关联，并在……时……】，不得影响【需要保持不变的现有流程】。本次明确不新增【预约、审核、支付、退款、消息提醒、时间冲突检测、库存变化、前台页面等已确认排除项】。后端代码必须生成到 `system.store.functionModule.【PascalName】Module`。本次【只生成后端及数据库 SQL/同时生成指定前端页面】，不生成【排除的代码范围】，不执行数据库脚本。

## 完整填空模板

```text
使用 $jx-build-backend-module，新增“【模块名称】”模块。

使用角色：【仅系统管理员 / 普通用户与管理员 / 指定角色】。

核心功能：【新增】、【修改】、【删除】、【分页查询】、【查看详情】；如有专用业务动作，填写【提交、审核、确认、完成等真实动作】。

查询条件：可按【条件一】、【条件二】、【条件三】筛选；如有默认排序，填写【排序依据】。

记录内容：包括【字段一】、【字段二】、【字段三】、【字段四】。

业务规则：【必填、唯一、数量范围、状态限制、重复提交限制或其他真实规则】。

业务关系：【本模块独立运行，不关联现有业务】；或【与现有“某模块”关联，关联用途是……】。

现有流程影响：【不得影响哪些现有模块、数据和流程】。

明确排除：【不需要预约、审核、支付、退款、消息、冲突检测、库存变化、导入导出等】。

代码路径：system.store.functionModule.【PascalName】Module。

生成范围：【仅后端和数据库 SQL / 后端加管理端页面 / 后端加用户端页面 / 指定范围】。

执行边界：【不执行数据库脚本 / 需要执行前另行确认】；【不启动服务 / 需要完成编译验证】。
```

## Skill 整理规则

1. 将用户已经表达的信息直接填入，不要求用户重复回答。
2. 根据中文模块名生成规范的 PascalName；存在多种合理英文命名且会影响路径时再确认。
3. 将“查询”细化为分页查询、详情查看和真实筛选条件，但不擅自增加导入、导出或统计。
4. 将普通维护与流程动作分开。申请、审核、发放、归还、结算、完成等必须有明确需求才能加入。
5. 明确模块是独立还是关联，不能用“暂不考虑”掩盖未确定的业务关系。
6. 明确前端、SQL、脚本执行、服务启动和验证范围，防止实施边界扩大。
7. 输出一段可直接交给 Skill 实施的完整需求；仍有关键未决项时，在需求段落后只列出必要的待确认项。

## 完整示例

使用 `$jx-build-backend-module`，新增“训练场地档案管理”模块，仅供系统管理员使用。管理员可以新增、修改、删除、分页查询和查看场地详情，并可按场地名称、场地类型和开放情况筛选。场地记录包括名称、类型、所在位置、容纳人数、联系人、联系电话、开放情况和说明。本模块独立运行，不与课程课时、比赛安排及其他现有业务建立关联，不新增预约、审核、时间冲突检测、消息提醒和前台页面。后端代码必须生成到 `system.store.functionModule.TrainingVenueModule`，本次只生成后端及数据库 SQL，不生成前端代码，不执行数据库脚本。
