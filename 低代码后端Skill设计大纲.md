# 低代码后端 Skill 设计大纲

> 当前阶段：设计大纲、正式 Skill、需求追踪、模块规格、代码生成细则、规则数据、只读检查工具、受保护脚手架、模板裁剪、模块验收、三类母版生成试验、Java 17 整合编译和最终回归均已完成；正式 Skill 已安装到个人技能目录，可进入实际项目使用与持续迭代阶段。
>
> 核心原则：大纲确定范围，细节定义规则，最终通过 Skill 快速生成和完善可编译验收的后台代码。

## 一、Skill 定位

### 1.1 建议名称

- 暂定名称：`jx-build-backend-module`
- 中文名称：低代码后端模块构建助手
- 最终名称在正式初始化 Skill 前确认。

### 1.2 核心目标

- 理解用户提出的后台业务需求。
- 以当前低代码平台真实代码和 KeyModule 模板为依据。
- 先形成模块大纲和业务边界，再生成或修改后台代码。
- 快速交付 Controller、Service、DAO、Entity、SQL 及业务所需的 DTO、VO 和 Mapper XML。
- 支持基础增删改查模块与复杂业务流程模块。
- 保证数据库、后台接口、权限和管理端调用保持一致。
- 通过编译、结构检查和业务不变量检查完成验收。

### 1.3 非目标

- 不替代需求分析，不在需求模糊时直接生成完整模块。
- 不把所有模块都强行套成基础增删改查。
- 不自动虚构审批、支付、退款、消息、二维码等业务。
- 不绕过现有 KeyModule 和 ModuleMaker 另建一套平行生成体系。
- 不在未经确认时修改公共框架、公共响应格式或权限规则。

## 二、事实来源与优先级

Skill 作出判断时，按照以下顺序获取事实：

1. 用户本次最新明确提出的修正、业务目标与限制。
2. 当前项目“按角色划分的功能描述”和“核心流程说明”等目标需求。
3. 当前项目中的数据库、实体、接口和实际业务代码。
4. `src/test/resources/KeyModule` 基础生成模板。
5. `ModuleMaker` 中的生成路径和名称替换规则。
6. 当前项目中已经运行的同类业务模块。
7. Skill 内置的项目母版、平台规则和通用后端经验。

发生冲突时，高优先级事实覆盖低优先级规则；Skill 内置知识不得覆盖项目真实实现。

### 2.1 业务模块默认归属

- 当前平台的业务模块默认统一归入 `system.store.functionModule.*`。
- 当前项目对应源码根目录为 `base-framework/src/main/java/system/store/functionModule`。
- Skill 检查其他同平台项目时，先根据项目源码根目录和 ModuleMaker 定位实际文件位置，但包路径仍以 `system.store.functionModule` 为默认根。
- 用户明确指定其他位置，或目标项目已经存在经过验证的不同结构时，才允许提出例外方案；不得静默改用其他包路径。

### 2.2 目标需求与当前实现的区分

- 用户提供的角色功能和核心流程用于定义项目要达到的目标状态。
- 项目代码、数据库和真实接口用于判断当前已经实现的状态。
- 目标需求存在、代码尚未支持时，标记为“待开发”，不能描述为“平台已有”。
- 代码存在、最新需求明确删除时，标记为“待移除或停用”，不能继续作为交付功能保留。
- 用户最新修正优先于较早版本的需求文档；存在冲突时记录变更，不自行选择旧版本。
- 项目母版只补充结构检查和常见风险，不能向当前项目自动增加需求中没有的菜单、流程或接口。

## 三、KeyModule 基础模板结论

### 3.1 模板文件构成

当前 KeyModule 由以下五类文件组成：

- `controller/KeyController.java`：详情、公开分页、管理分页、新增、修改、逻辑删除、恢复、批量状态、批量排序、元数据、Excel 导入导出等接口与权限入口。
- `service/KeyService.java`：分页过滤、创建人数据范围、基础维护、逻辑删除恢复、批量操作、数据校验、操作日志和低代码元数据等业务。
- `dao/KeyDao.java`：基于 MyBatis-Plus 的数据访问入口。
- `model/entity/Key.java`：实体、逻辑删除、数据表映射、前端生成标记和 Excel 字段说明。
- `db/c_key_table.sql`：基础建表、菜单与按钮权限注册脚本。

### 3.2 ModuleMaker 生成规则

- 模板来源：`src/test/resources/KeyModule`。
- 后台模块目标：`src/main/java/system/store/functionModule`。
- 模块目录：以大驼峰英文名称加 `Module` 组成。
- `Key`：替换为大驼峰模块名。
- `key`：替换为小驼峰模块名。
- 下划线名称：用于数据库表名和 SQL 文件名。
- `XX`：替换为模块中文名称。
- 目标目录已经存在时停止生成，避免覆盖现有模块。

### 3.2.1 业务模块目录与包路径强制规则

- 新业务模块默认文件目录：`src/main/java/system/store/functionModule/<大驼峰模块名>Module`。
- 当前仓库的完整位置：`base-framework/src/main/java/system/store/functionModule/<大驼峰模块名>Module`。
- 新业务模块默认包根：`system.store.functionModule.<大驼峰模块名>Module`。
- 控制入口、业务服务、数据访问和业务模型继续位于该模块目录内部，不拆到平行的业务根目录。
- 子目录与包声明必须逐级对应，例如服务文件位于 `.../<模块名>Module/service`，包声明对应 `system.store.functionModule.<模块名>Module.service`。
- 模块之间的引用使用上述完整包路径，不通过重复创建同名类规避引用问题。
- `functionModule` 的大小写属于平台约定，不能写成其他大小写形式。
- 系统级用户、角色、菜单等公共能力继续复用既有系统模块，不强行迁入 `functionModule`。
- 发现同名模块已经位于其他位置时，先报告差异和引用影响，不自动移动、不重复生成。

### 3.3 基础模板能够保证的内容

- 标准模块目录。
- 公开详情、公开分页与后台管理分页入口。
- 基础新增、修改、批量逻辑删除和删除恢复。
- 批量状态与排序入口。
- 标准权限命名结构。
- 统一分页与响应封装。
- 基础实体和数据访问结构。
- 创建人数据范围、通用筛选、基础校验与操作日志位置。
- 低代码元数据、Excel 导入导出入口。
- 基础建表与菜单按钮权限脚本。

### 3.4 基础模板不能代表已经完成的内容

- 业务状态流转。
- 多表关联与级联校验。
- DTO、VO 和专用查询条件。
- 库存、核销、日志等复合业务。
- 多步骤事务。
- 用户数据权限。
- 唯一性、并发和重复提交控制。
- 与前端实际页面完全匹配的查询和表单配置。
- 复杂联表查询所需的自定义 Mapper XML。

因此，Skill 必须明确区分“基础骨架生成”和“业务增强”两个阶段。

### 3.5 模板能力开关

ModuleMaker 会复制当前 KeyModule 的完整骨架，但 Skill 不能因此默认启用所有能力。生成前必须形成开关清单：

| 模板能力 | 默认处理 | 启用条件 |
|---|---|---|
| 后台详情与分页 | 保留 | 后台需要查看和维护该资料 |
| 新增、修改、逻辑删除 | 按模块确认 | 统计只读、流水只读等模块可关闭部分写入 |
| 公开详情与公开分页 | 默认关闭 | 明确存在无需后台权限的用户端展示 |
| 删除恢复 | 默认关闭 | 业务要求后台恢复逻辑删除数据 |
| 批量状态 | 默认关闭 | 实体存在通用启停状态且允许批量维护 |
| 批量排序 | 默认关闭 | 实体存在排序字段且页面提供排序操作 |
| 低代码元数据 | 默认保留 | 当前管理端继续使用元数据生成或适配能力 |
| Excel 导入 | 默认关闭 | 用户确认批量导入，字段和重复数据规则已经明确 |
| Excel 导出 | 默认关闭 | 用户确认导出范围、字段和数据权限 |
| 菜单与按钮权限 SQL | 与保留接口同步 | 只注册实际存在的页面和按钮能力 |

关闭能力时，必须同时处理 Controller、Service、实体字段说明、权限 SQL 和管理端调用，不能只隐藏页面按钮。

### 3.6 自定义 Mapper 使用边界

- 基础单表查询优先使用现有 MyBatis-Plus 能力，不强制生成 Mapper XML。
- 多表联查、复杂聚合或现有项目已经采用 XML 查询时，才增加 `src/main/resources/mapper/function` 下的 Mapper XML。
- DAO 中的自定义方法、Mapper XML 的命名空间、参数和返回结构必须逐项对应。
- DTO、VO 或聚合结果只用于真实查询与交互需要，不为凑齐目录而生成空类。

## 四、总体工作阶段

### 阶段一：需求理解

- 明确要创建新模块、增强旧模块、排查问题还是仅输出设计。
- 提取业务对象、参与角色、核心动作和最终结果。
- 标记用户明确禁止或暂不需要的能力。
- 找出会影响数据结构和流程的关键歧义。

### 阶段二：项目事实检查

- 定位 KeyModule、ModuleMaker 和目标模块。
- 将 `system.store.functionModule` 作为默认业务包根，核对目录位置、包声明和引用是否一致。
- 检查现有实体、SQL、接口、服务、前端调用和权限。
- 查找最接近的真实业务模块作为补充依据。
- 识别工作区已有改动，避免覆盖用户代码。

### 阶段三：先输出模块大纲

大纲至少包含：

- 模块目标。
- 角色与权限边界。
- 核心数据对象。
- 基础增删改查范围。
- 专用业务动作。
- 状态流转。
- 数据关联。
- 事务边界。
- 前后端契约。
- 验收方式。
- 明确不实现的内容。

### 阶段四：确认内容边界

- 核对大纲是否覆盖用户真实需求。
- 确认模块属于基础型还是业务增强型。
- 确认哪些字段来自数据库，哪些信息只用于展示。
- 确认哪些操作需要管理员、普通用户或公共访问权限。
- 确认是否允许创建新表、修改旧表或增加接口。

### 阶段五：细化设计

- 形成字段清单和数据关系。
- 形成接口清单和请求响应结构。
- 形成状态转换表。
- 形成校验规则与错误反馈。
- 形成事务、库存、日志和并发处理策略。
- 形成需要生成、修改和保留的文件清单。

### 阶段六：生成或修改代码

- 基础模块优先复用 KeyModule 与 ModuleMaker。
- 在生成骨架上完成业务增强。
- 使用当前项目已有工具类、分页、响应和权限结构。
- 仅修改需求范围内文件。

### 阶段七：验收

- 检查目录和命名。
- 检查实体与 SQL 一致性。
- 检查权限和前端入口一致性。
- 检查状态转换和业务不变量。
- 执行后台编译和必要测试。
- 对多表写入、库存变化和核销流程进行专项验证。

## 五、模块分类体系

### 5.1 基础资料型

- 以单表增删改查为主。
- 可以直接使用 KeyModule 骨架。
- 重点检查字段、查询条件、校验和权限。

### 5.2 关联资料型

- 存在外键、字典、用户或其他模块关联。
- 需要显示名称而非只显示关联编号。
- 通常需要专用查询或 VO。

### 5.3 状态流程型

- 存在申请、确认、完成、取消等状态变化。
- 必须先定义允许的状态转换。
- 禁止使用通用修改接口任意改变关键状态。

### 5.4 库存核销型

- 涉及库存、数量、发放、归还、报损或核销。
- 必须定义数量守恒关系、事务边界和流水记录。
- 必须防止超量、重复确认和库存负数。

### 5.5 统计分析型

- 以聚合查询和可视化数据接口为主。
- 不应强行生成无意义的新增、修改和删除入口。
- 重点检查口径、时间范围和空数据处理。

### 5.6 系统配置型

- 涉及用户、角色、菜单、字典和系统配置。
- 必须优先复用现有系统模块。
- 不得另建重复的权限或用户体系。

## 六、Skill 的输出模式

### 6.1 分析模式

- 只检查现有模块并说明真实流程、问题和风险。
- 不修改代码。

### 6.2 项目蓝图模式

- 根据毕设方向、角色、使用端和难度形成候选母版与模块组合。
- 明确区分候选能力、平台已有能力和待开发能力，不生成正式代码。

### 6.3 大纲模式

- 输出模块设计大纲、范围和待确认项。
- 不生成正式代码。

### 6.4 基础生成模式

- 按 KeyModule 规则创建基础模块骨架。
- 仅适用于范围已经确认的基础资料型模块。

### 6.5 业务增强模式

- 在现有骨架上增加 DTO、VO、状态动作、事务和专用校验。
- 必须以已确认的大纲和细化设计为输入。

### 6.6 审计修复模式

- 检查现有模块是否符合平台规范和业务不变量。
- 只有用户明确要求修复时才修改代码。

## 十、Scripts 职责边界

### 10.1 inspect-platform.py

- 定位 KeyModule、ModuleMaker、后台模块目录和前端模块目录。
- 确认 `system.store.functionModule` 的实际源码根、现有模块数量和异常分布。
- 输出平台能力清单，不修改文件。

### 10.2 inspect-module.py

- 盘点目标模块的实体、SQL、接口、服务、权限和前端调用。
- 检查目标模块是否位于 `system.store.functionModule.<模块名>Module`，以及目录、包声明和引用是否一致。
- 输出缺失项和冲突项，不直接修复。

### 10.3 validate-project-requirements.py

- 检查角色功能清单与核心流程是否相互覆盖、角色一致、闭环完整。
- 输出遗漏功能、孤立流程步骤、角色冲突、待开发能力和需要确认的歧义。
- 只生成需求覆盖报告，不根据常见母版自动补充需求。

### 10.4 validate-module-spec.py

- 检查模块规格中的命名、字段、查询、权限、状态、事务和能力开关是否完整。
- 输出可生成项、缺失项和阻断项，不修改项目文件。

### 10.5 search-rules.py

- 根据模块类型、风险和动作检索规则。
- 可以借鉴 UI/UX Pro Max 的轻量检索机制。

### 10.6 build-module-outline.py

- 根据已检查的项目事实生成结构化大纲草案。
- 输出必须允许人工修改和确认。

### 10.7 build-project-blueprint.py

- 根据题目方向、角色、使用端和复杂度生成候选项目母版。
- 输出必选模块、可选模块、主流程、风险和待确认项。
- 只形成方案草稿，不声称候选模块已经实现。

### 10.8 scaffold-module.py

- 根据确认后的四种名称形成 ModuleMaker 生成计划并默认预演。
- 预演结果必须指向 `system.store.functionModule.<模块名>Module` 对应目录。
- 目标目录存在、内容未确认或未获实施授权时禁止写入。

### 10.9 validate-module.py

- 检查实体与 SQL。
- 检查接口与前端服务。
- 检查权限命名。
- 检查业务模块目录、包声明和跨模块引用是否统一以 `system.store.functionModule` 为根。
- 检查状态动作是否绕过专用入口。
- 输出问题清单，不擅自修改代码。

## 十二、强制阶段门禁

### 门禁一：大纲确认

未形成完整项目蓝图或模块大纲前，不进入字段和代码细化。

### 门禁二：内容确认

以下内容没有确认时，不生成正式代码：

- 模块目标。
- 角色权限。
- 数据对象。
- 专用业务动作。
- 状态流程。
- 数据库修改范围。

规划整套项目时，还必须确认主业务大类、使用端形态、必选模块、可选增强和明确不做事项。

### 门禁三：实施授权

只有用户明确要求创建或修改时，才写入项目文件。

### 门禁四：验收完成

编译通过不等于完成；必须同时完成结构、权限、数据和业务闭环检查。

## 十三、反脑补与安全红线

- 不根据模块名称猜测不存在的业务动作。
- 不根据前端按钮猜测后台一定存在对应能力。
- 不因为数据库存在字段就推断页面已经支持相关操作。
- 不使用通用编辑接口代替关键状态动作。
- 不在申请阶段提前执行发放、扣减、核销等最终动作。
- 不把未经核验的前端身份或数量作为可信事实。
- 不删除或覆盖已经存在的目标模块。
- 不修改与当前任务无关的公共框架。
- 不将相似模块的业务规则直接复制到新模块。
- 不把新业务模块创建到 `system.store.functionModule` 之外的平行业务目录。
- 不通过修改包名大小写、创建重复同名模块或散落业务类来绕过标准目录约束。

## 十四、验收矩阵大纲

| 验收维度 | 核心问题 |
|---|---|
| 需求覆盖 | 每条角色功能是否有实现状态、代码去向和验收结果 |
| 流程追踪 | 每个核心流程步骤是否有校验、数据、状态和异常去向 |
| 项目选型 | 主业务大类、使用端和母版组合是否与题目一致 |
| 能力状态 | 候选、已确认、平台已有、待开发和已验证是否有事实依据 |
| 目录结构 | 是否位于 `system.store.functionModule.<模块名>Module` 并符合 KeyModule 结构 |
| 包路径 | 目录、包声明、导入引用是否统一以 `system.store.functionModule` 为业务根 |
| 命名 | 中文名、大驼峰、小驼峰、下划线名是否一致 |
| 数据库 | SQL、表名、字段和实体是否一致 |
| 接口 | 路径、请求、响应和前端服务是否对应 |
| 权限 | 页面权限和按钮权限是否与操作一致 |
| 查询 | 分页、关联名称、查询条件是否真实有效 |
| 写入 | 新增、修改、删除是否有必要校验 |
| 状态 | 每个动作是否只允许在合法状态执行 |
| 事务 | 多项数据变化能否同时成功或同时失败 |
| 数量 | 库存、归还、报损等数量是否满足守恒关系 |
| 日志 | 关键业务是否留下可追溯记录 |
| 构建 | 后台是否编译，相关前端是否构建通过 |
| 闭环 | 发起、确认、完成和异常是否都有最终去向 |

## 十五、首版范围推荐决策

以下七项先作为首版默认决策写入内容稿。后续如果使用中发现不合适，可在正式初始化前调整，不影响当前大纲结构。

| 决策项 | 首版推荐结论 | 原因 |
|---|---|---|
| Skill 名称 | `jx-build-backend-module` | 名称直接表达“为当前平台构建后端模块”的用途 |
| 创建位置 | 先在项目内评审，稳定后再安装到个人技能目录 | 方便与平台模板共同校验，也避免未成熟版本影响其他项目 |
| 技术范围 | 首版只支持当前 Spring Boot、MyBatis-Plus 与既有公共框架 | 先保证规则真实、结果稳定，不做跨框架泛化 |
| 前端范围 | 检查管理端 Vue 调用契约，不承担页面视觉设计 | 后端变更需要验证页面能否真实调用，但不与 UI 技能职责重叠 |
| 模板生成 | 允许使用现有 ModuleMaker，但必须经过内容确认和实施授权 | 复用平台真实生成方式，同时保留防误写门禁 |
| 验收样例 | KeyModule、CourseOrderModule、EquipmentClaimModule | 分别覆盖基础骨架、关联业务、状态与库存闭环 |
| 复杂样例 | 装备领取作为复杂流程样例，但只提炼通用规则 | 借鉴数量守恒和状态校验，不复制该模块的具体业务 |

### 15.1 首版能力边界

- 支持毕设项目母版规划、新模块设计、基础骨架生成、现有模块增强、现有模块审计五类任务。
- 支持基础资料、关联查询、状态流程、库存核销、统计分析和系统配置六类模块识别。
- 支持从常见题目方向形成候选模块组合，但正式实施仍按单个模块逐项确认。
- 支持数据库、后台接口、业务服务、权限和管理端调用契约的联合检查。
- 不支持自动设计新的公共框架、用户体系、权限体系或前端视觉规范。
- 不把其他项目的目录、命名和业务规则自动套用到当前平台。
- 不把项目母版、可复用能力目录或典型流程描述成当前仓库已经实现的功能。

## 十六、正式 Skill 的触发与定位文案

### 16.1 建议触发描述

当用户提供按角色划分的功能描述、核心流程、毕设项目需求或现有项目代码，需要为当前低代码平台规划项目、创建模块、增强流程、审查实现或快速完成后台代码时使用。Skill 必须将项目需求转换为功能覆盖矩阵与流程到代码追踪矩阵，区分目标需求和当前实现，将 `system.store.functionModule.*` 作为业务模块默认归属；先检查 KeyModule、ModuleMaker、目标模块和相邻真实模块，经确认后生成或修改 Controller、Service、DAO、实体、SQL 及必要的 DTO、VO、Mapper XML，并联合验证目录包路径、权限、事务、状态、数量守恒和管理端调用契约。

### 16.2 不应触发的场景

- 只做登录页、表格、配色、布局等前端视觉优化。
- 只编写与当前低代码平台无关的通用 Java 示例。
- 只整理产品文档且不需要核对后台真实实现。
- 用户明确要求使用另一套框架或另一套生成体系。

### 16.3 建议展示信息

- 展示名称：低代码后端模块构建助手。
- 简短说明：基于平台真实模板，先定业务边界，再生成、增强和验收后台模块。
- 默认任务提示：读取项目的角色功能清单与核心流程，区分目标需求和当前实现，以 `system.store.functionModule.*` 为默认业务包根，先形成需求覆盖和流程追踪结果；规格完整且已获实施授权后，生成并验收完整后台代码。

## 十七、任务输入契约

Skill 收到请求后，不要求用户一次性提供所有技术细节，但必须整理出以下内容。能够从项目中查明的内容直接查明；会改变业务结果且无法从项目判断的内容，列为待确认项。

### 17.1 必填业务信息

| 信息 | 内容要求 |
|---|---|
| 任务类型 | 项目母版规划、新建模块、增强模块、审查模块或解释流程 |
| 业务目标 | 该模块要解决什么问题，最终形成什么结果 |
| 参与角色 | 谁可以查看、发起、处理、确认或结束业务 |
| 核心对象 | 需要管理的主要资料、记录及其相互关系 |
| 核心动作 | 除基础查询维护外，真实存在的业务操作 |
| 禁止能力 | 用户明确不需要、项目没有实现或暂不纳入的内容 |

### 17.2 条件必填信息

- 存在状态流转时，必须明确起点、允许动作、结果状态和终止条件。
- 存在库存、名额、积分或次数时，必须明确增加、扣减、退回、核销和异常处理规则。
- 存在多角色数据范围时，必须明确本人、本队、本课程或全部数据的边界。
- 需要修改数据库时，必须明确允许新增表、增加字段还是仅使用现有结构。
- 需要生成新模块时，必须明确中文名、大驼峰名、小驼峰名和下划线名。
- 规划整套毕设项目时，必须明确题目方向、主要角色、使用端形态、必选模块和期望难度。

### 17.3 项目自动发现信息

- 项目根目录、后台模块目录、管理端模块目录。
- `system.store.functionModule` 对应的源码根、现有模块分布和包声明一致性。
- KeyModule 与 ModuleMaker 的真实位置和当前内容。
- 目标模块是否已经存在，工作区是否已有相关修改。
- 公共响应、分页、异常、登录身份和权限的既有用法。
- 最接近的基础模块、关联模块和复杂业务模块。
- 数据库脚本、实体、接口、业务服务、管理端调用是否相互对应。

### 17.4 信息不足时的处理

- 不影响业务结果的命名和排版细节，可采用平台既有规则并明确说明。
- 能通过项目文件确认的事实，不重复询问用户。
- 会改变角色权限、数据结果、状态去向或库存数量的歧义，必须停止实施并请求确认。
- 用户只要求分析或设计时，不把请求扩大为代码修改。

### 17.5 标准项目需求输入格式

以后开发新项目时，Skill 优先接收以下三部分内容，不要求用户先写技术字段：

1. **按角色划分的功能描述**：按学员、教练、管理员或当前项目实际角色，逐条提供“功能名称：业务说明”。
2. **核心流程说明**：用完整段落描述谁发起、线下如何配合、系统校验什么、数据如何变化、状态如何结束。
3. **最新修正与禁止事项**：明确需要删除、暂缓或绝对不能新增的功能。

项目名称、技术范围和目标代码位置能够从当前项目或 Skill 中查明时无需重复提供；无法查明且会影响结果时再集中询问。

### 17.6 角色功能标准化

每条角色功能必须转换为一条可追踪需求，至少包含：

| 内容 | 说明 |
|---|---|
| 需求编号 | 保证后续代码和验收能够反向定位原始需求 |
| 角色 | 谁可以使用该能力 |
| 功能名称 | 菜单、页面或业务能力名称 |
| 业务目标 | 用户通过该功能完成什么结果 |
| 动作类型 | 查看、维护、提交、审核、确认、统计或专用动作 |
| 数据范围 | 本人、本课程、本队伍、指定范围或全部数据 |
| 关联模块 | 依赖哪些资料、记录或公共能力 |
| 覆盖状态 | 平台已有、部分实现、待开发、待移除或已验证 |

同一后台能力被多个角色使用时，可以复用 Service 和数据查询，但必须分别保留角色权限、数据范围与入口差异。

### 17.7 核心流程标准化

每条核心流程拆分为连续步骤，每一步至少记录：流程编号、步骤顺序、执行角色、线上操作、线下配合、前置条件、系统校验、数据变化、结果状态、异常去向和负责模块。

- 线上操作必须能映射到真实页面或接口。
- 线下配合只描述现实动作，不转化为系统自动能力。
- 系统校验必须在 Service 或数据库约束中有落实位置。
- 数据变化必须能够映射到实体、SQL、事务或流水记录。
- 状态变化必须使用专用业务动作，不能依赖通用编辑猜测完成。
- 正常完成、拒绝、取消、超量、重复或失败等已确认分支必须有最终去向。

### 17.8 需求与流程交叉检查

- 每个核心流程步骤至少由一项角色功能承载。
- 每项进入开发范围的角色功能至少映射到一个后台模块或公共能力。
- 功能清单出现、核心流程未覆盖的关键动作，列为补充流程或待确认项。
- 核心流程出现、功能清单未出现的操作，列为遗漏功能，不能静默新增菜单。
- 不属于后台代码的纯展示或线下动作仍保留业务说明，但标记为无需新增后台模块。
- 同名功能在不同角色下含义不同时分别建需求记录，不能仅按名称合并。

## 十八、任务输出契约

### 18.1 大纲阶段输出

每次进入实现前，至少给出以下内容：

1. 模块目标与不做事项。
2. 角色和数据范围。
3. 核心对象与关联关系。
4. 基础维护能力与专用业务动作。
5. 状态流程或数量变化规则。
6. 数据库、后台和管理端的预计影响文件。
7. 风险、待确认项和验收方式。

### 18.2 内容确认阶段输出

内容稿使用固定的六张清单，保证后续实现有明确依据：

| 清单 | 必须回答的问题 |
|---|---|
| 数据清单 | 保存什么、关联什么、哪些内容必须唯一或不能为空 |
| 动作清单 | 谁在什么前提下执行什么动作，成功后改变什么 |
| 状态清单 | 哪些状态允许转换，哪些状态属于终态 |
| 权限清单 | 页面、按钮、公共查询和本人数据分别如何控制 |
| 接口清单 | 页面实际需要哪些查询和操作，输入输出是什么 |
| 验收清单 | 正常、重复、越权、超量、空数据和失败回滚如何验证 |

### 18.3 实施阶段输出

- 先列出将创建和修改的文件，再实施已确认范围。
- 新业务模块默认落入 `system.store.functionModule.<模块名>Module`，并在文件清单中明确完整目录和包路径。
- 新建基础模块时说明哪些文件来自模板，哪些属于业务增强。
- 修改现有模块时只改任务相关内容，保留用户已有改动。
- 完成后报告编译、结构、契约和业务闭环的验证结果。
- 无法验证的项目必须明确标注，不能用“应该可用”代替结果。

### 18.4 审查阶段输出

- 按严重程度列出问题、事实依据、业务影响和建议修复方向。
- 区分已确认缺陷、潜在风险和尚需业务确认的歧义。
- 用户未要求修复时只做检查，不直接写入文件。

### 18.5 项目级需求追踪输出

当用户提供完整角色功能和核心流程时，实施前额外输出两张矩阵：

**功能覆盖矩阵**至少包含：需求编号、角色、功能、目标模块、实现状态、Controller 入口、Service 处理、数据对象、权限和验收结果。

**流程到代码追踪矩阵**至少包含：流程步骤、执行角色、线上或线下动作、校验位置、数据变化、状态变化、事务边界、异常去向和对应文件。

矩阵用于保证：

- 用户列出的功能没有被遗漏。
- 代码新增的接口能够找到明确需求依据。
- 核心流程中的每次数据与状态变化都有实现位置。
- 需求被删除或调整后，相关接口、权限和 SQL 能够同步定位。
- 最终交付可以从需求反查代码，也可以从代码反查业务来源。

## 十九、规则数据文件内容定义

规则库只保存可重复使用、能够明确检查的短规则；需要上下文解释的内容放入 references，不把全部平台知识塞进表格。

### 19.1 module-types.csv

建议列：规则编号、模块类型、识别信号、必需组成、必查风险、禁止默认行为、参考模块。

- 基础资料型参考 KeyModule，只保证标准骨架，不默认增加复杂流程。
- 关联业务型参考 CourseOrderModule，重点检查关联对象、重复业务和角色数据范围。
- 库存核销型参考 EquipmentClaimModule，重点检查动作入口、事务和数量守恒。

### 19.2 permission-rules.csv

建议列：规则编号、访问范围、接口类别、权限结构、身份来源、数据范围检查、风险等级。

- 页面和按钮权限沿用当前平台命名结构。
- 本人数据从当前登录身份取得，不信任页面提交的身份。
- 公共接口必须有明确业务理由，不能仅为省略权限配置而开放。

### 19.3 endpoint-patterns.csv

建议列：规则编号、动作类别、适用模块、请求方式、路径结构、输入形式、输出形式、权限要求、禁止用法。

- 基础详情、分页、新增、修改和删除遵循平台现有结构。
- 发放、结课、归还、报损等动作使用专用入口，不借用通用编辑接口改变关键状态。
- 页面未实际使用、业务未确认的接口不自动生成。

### 19.4 field-rules.csv

建议列：规则编号、业务含义、数据类型、是否必填、取值约束、默认来源、展示说明、数据库要求。

- 字段规则从业务含义出发，再映射实体与数据结构。
- 状态、数量和关联标识必须写清取值来源及合法范围。
- 不因模板中存在示例字段就默认保留到新模块。

### 19.5 transaction-rules.csv

建议列：规则编号、触发动作、涉及记录、必须同时完成的变化、失败条件、回退要求、重复操作处理。

- 同一业务动作同时改变主记录、库存和日志时，必须作为一个整体完成。
- 任一前置校验失败时，不产生部分库存变化或半完成状态。
- 重复确认不能造成重复扣减、重复恢复或重复奖励。

### 19.6 state-rules.csv

建议列：规则编号、业务对象、当前状态、允许动作、执行角色、前置条件、结果状态、伴随变化、禁止动作。

- 每个关键动作只能从明确允许的状态进入。
- 终态默认不允许通过通用修改回到处理中。
- 同一记录存在部分完成时，必须定义剩余数量和后续允许动作。

### 19.7 validation-rules.csv

建议列：规则编号、校验场景、检查对象、通过条件、失败反馈、检查层次、测试示例。

- 同时覆盖空值、范围、重复、关联存在性、身份、状态和数量。
- 页面校验用于及时提示，后台校验负责保证最终业务安全。
- 错误反馈说明业务原因，不暴露内部实现细节。

### 19.8 anti-patterns.csv

建议列：规则编号、错误做法、识别信号、业务风险、替代方式、适用范围。

首批至少收录：仅复制 KeyModule 就宣称流程完成、通用编辑直接改状态、申请时提前扣库存、只信任页面身份、只编译不验业务、覆盖现有模块、虚构未实现接口、统计接口返回无依据模拟数据。

### 19.9 project-archetypes.csv

建议列：大类编号、业务大类、细分方向、题目命名参考、常见主流程、典型必选能力、可选增强能力、默认复杂度、关键确认项、风险提示。

- 记录“常见做法”而不是“当前系统事实”。
- 一个题目可以命中多个大类，但必须确定一个主业务闭环。
- 小程序和 APP 记录为使用端形态，不与业务大类混为一谈。

### 19.10 reusable-capabilities.csv

建议列：能力编号、能力名称、业务职责、适用题型、前置依赖、典型数据、典型动作、验收重点、默认是否启用、风险提示。

- 用户、管理员和基础信息维护可作为高频基础能力。
- 审核、订单、评论、推荐和统计均按题目选择，不自动全量启用。
- 平台已有能力标记为“复用”，只有设计没有代码的标记为“候选”。

### 19.11 solution-presets.csv

建议列：母版编号、母版名称、适用题型、必选能力、可选能力、主流程、关键状态、实施顺序、最低验收、禁止默认项。

- 母版是模块组合方案，不是可直接复制的代码目录。
- 母版选中后仍需逐模块完成大纲、内容确认和项目事实检查。
- 可选能力只有用户明确选择后才进入实施范围。

### 19.12 template-capabilities.csv

建议列：能力编号、模板入口、涉及文件、默认开关、启用条件、依赖字段、权限要求、裁剪检查、验收方式。

- 首批覆盖公开查询、基础维护、恢复、批量状态、批量排序、元数据、Excel 导入和 Excel 导出。
- 开关关闭时列出必须同步删除或调整的 Controller、Service、权限 SQL 和前端调用。

### 19.13 java-sql-types.csv

建议列：业务类型、Java 类型、数据库类型、长度精度、默认值规则、空值规则、常用注解、适用限制。

- 用于快速形成实体与 SQL 的一致映射。
- 日期、金额、长文本、布尔状态和关联标识必须保留业务语义，不只按示例字段猜类型。

### 19.14 query-patterns.csv

建议列：查询编号、查询类型、适用场景、Service 实现方式、是否需要自定义 DAO、是否需要 Mapper XML、分页方式、空数据处理、验收要点。

- 覆盖精确筛选、模糊搜索、时间范围、关联名称、聚合统计和排行等常见查询。
- 单表能力足够时不生成 Mapper XML；复杂联表与聚合才进入自定义查询。

### 19.15 requirement-mapping-rules.csv

建议列：规则编号、需求表达类型、识别信号、目标代码层、权限影响、数据影响、流程影响、必须确认项、覆盖检查、禁止推断。

- 角色端的“查看、维护、提交、审核、确认、统计”等业务动词映射为不同代码职责，不能全部转换为通用 CRUD。
- 线下交接、现场训练、实物验收等步骤只记录为业务前提，不虚构二维码、定位、支付或自动感知能力。
- 同一功能被多个角色使用时，优先复用业务服务，并分别定义接口权限与数据范围。

## 二十、检查与生成工具契约

首版工具优先做“检查”和“形成草稿”，自动写入能力最后开放。所有工具都接受项目根目录参数，默认输出可阅读报告，并支持结构化结果供后续工具使用。

### 20.1 inspect-platform.py

- 输入：项目根目录、可选输出格式。
- 输出：模板位置、生成器位置、`system.store.functionModule` 实际源码根、后台和管理端目录、公共能力、发现的问题。
- 约束：只读，不创建目录，不修改配置。

### 20.2 inspect-module.py

- 输入：项目根目录、目标模块名、可选相邻参考模块。
- 输出：现有文件、标准包路径匹配结果、接口、权限、数据结构、管理端调用、工作区冲突和缺失项。
- 约束：只读；目标不存在时明确报告，不自动生成。

### 20.3 validate-project-requirements.py

- 输入：按角色划分的功能描述、核心流程说明、最新修正和项目检查结果。
- 输出：功能覆盖矩阵、流程追踪矩阵、需求与当前实现差异、冲突和待确认项。
- 约束：只检查用户给出的需求，不使用项目母版静默添加功能。

### 20.4 validate-module-spec.py

- 输入：模块规格文件、项目检查结果和生成档位。
- 输出：命名、字段、查询、权限、状态、事务、能力开关的通过项与缺失项。
- 约束：存在影响代码结果的缺失项时阻止生成，不使用猜测值自动补齐。

### 20.5 search-rules.py

- 输入：模块类型、业务动作、风险关键词和返回数量。
- 输出：命中的规则、命中原因、对应参考文档。
- 约束：检索结果只作为检查提示，不能覆盖项目事实。

### 20.6 build-project-blueprint.py

- 输入：题目方向、角色、使用端形态、复杂度、必选和禁止能力。
- 输出：候选母版、必选模块、可选模块、主流程、实施顺序、风险和待确认项。
- 约束：母版只表示建议组合；未经项目检查不得标记为已实现。

### 20.7 build-module-outline.py

- 输入：平台检查结果、模块检查结果和已确认业务信息。
- 输出：符合第十八章结构的大纲草稿及待确认项。
- 约束：只生成文档内容，不生成正式代码。

### 20.8 scaffold-module.py

- 输入：项目根目录、四种模块名称、中文名和已确认标记。
- 输出：ModuleMaker 计划、`system.store.functionModule.<模块名>Module` 目标包、目标目录和预期文件清单。
- 约束：首版默认预演；只有目标不存在、内容已确认且获得实施授权时才执行生成。

### 20.9 validate-module.py

- 输入：项目根目录、目标模块、验证范围。
- 输出：目录与包路径、引用关系、结构、业务规则的通过项、警告项、阻断项、依据文件和建议处理方向。
- 约束：默认只读，不自动修复；编译成功不能抵消业务阻断项。

### 20.10 统一结果级别

- 通过：已有直接证据，符合平台规则和已确认业务。
- 警告：不一定错误，但需要人工判断或补充测试。
- 阻断：会导致编译失败、权限越界、数据错误、非法状态或数量失真。
- 未验证：受运行环境、缺少数据或外部条件限制，不能得出结论。

对于新建业务模块，目标不在 `system.store.functionModule.<模块名>Module`、目录与包声明不一致、或出现同名重复模块时，默认列为阻断项；只有用户确认的项目级例外才能降级处理。

## 二十一、首版验收样例

### 21.1 样例一：KeyModule 基础骨架

验证 Skill 能否准确识别五类模板文件、四种命名替换、`system.store.functionModule.<模块名>Module` 目标归属、目录包声明一致性、目标目录保护、标准接口、统一响应和权限结构。验收重点是“忠实复用模板”，不要求凭空增加关联查询、状态流程或统计功能。

### 21.2 样例二：CourseOrderModule 关联业务

验证 Skill 能否识别报名记录与学员、课程、教练和课时之间的关联，区分学员本人数据与教练负责课程的数据范围，识别重复报名校验、报名详情组合信息和结课动作。验收重点是“基础骨架之外必须根据真实业务补充专用查询和校验”。

### 21.3 样例三：EquipmentClaimModule 复杂闭环

验证 Skill 能否识别申请、发放、归还申请、线下验收、正常归还、损坏或丢失处理等专用动作；检查发放时扣减可用数量、正常归还时恢复可用数量、报损丢失时调整装备总量，以及重复操作、超量和非法状态是否被阻止。验收重点是“状态、数量、事务和日志形成完整闭环”。

### 21.4 通用测试任务

| 测试任务 | 预期结果 |
|---|---|
| 只要求分析现有流程 | 读取真实实现并输出依据，不修改任何文件 |
| 新建基础资料模块 | 先给大纲，经确认后按模板生成并完成契约检查 |
| 增强关联业务模块 | 保留现有代码，只补充已确认的关联查询和校验 |
| 增强库存状态模块 | 先定义状态与数量守恒，再实施专用动作和事务 |
| 目标目录已经存在 | 停止自动生成，转为检查或增量修改方案 |
| 工作区存在用户改动 | 标明重叠范围并保留无关改动，不覆盖或回退 |
| 用户提出项目不存在的能力 | 标记为新增需求，不能描述成当前已实现功能 |

## 二十二、首版完成标准

### 22.1 文档层

- 正式 SKILL.md 保持精炼，核心流程、门禁和路由清晰。
- references 每个文件职责单一，避免重复描述同一规则。
- 规则表字段固定，首批规则都有事实依据或明确适用范围。

### 22.2 工具层

- 所有检查工具在目标不存在、输入不全和路径异常时给出明确结果。
- 平台检查能够定位 `system.store.functionModule` 实际源码根，模块检查能够发现错误包路径和散落业务类。
- 只读工具不会修改项目时间戳、配置或源文件。
- 生成工具默认预演，并对已存在目录执行硬性保护。
- 验证结果能够区分通过、警告、阻断和未验证。

### 22.3 业务层

- 不生成用户未确认的角色、状态、接口和数据字段。
- 基础模块、关联模块和复杂模块三类样例均能得到正确结论。
- 权限、事务、状态和数量检查能够发现真实问题，而不是只检查目录是否齐全。
- 输出能够说明哪些内容来自模板、哪些来自项目事实、哪些仍需用户确认。

## 二十三、版本推进顺序

### 0.1 内容版

- 创建正式目录、SKILL.md、展示信息、平台规则和项目母版知识库。
- 先以人工检查流程运行，不急于开发生成工具。

### 0.2 检查版

- 完成平台检查、模块检查、母版检索、规则检索和统一报告结构。
- 用三个验收样例校正规则，优先消除误报和漏报。

### 0.3 辅助生成版

- 完成项目蓝图、大纲草稿工具和基础骨架预演。
- 在门禁全部满足后，允许调用现有 ModuleMaker。

### 1.0 稳定版

- 基础模块、关联模块和复杂闭环模块均通过回归验收。
- 项目事实优先级、工作区保护和反脑补规则稳定。
- 再安装到个人技能目录，供同类低代码项目按明确适配规则使用。

### 暂缓内容

- 适配其他 Java 框架或其他低代码平台。
- 自动设计前端视觉样式。
- 自动执行数据库迁移或生产数据修复。
- 自动修改公共用户、角色和菜单体系。
- 无人工确认的一键生成完整复杂业务模块。

## 二十四、毕设项目知识库分层

这批常见毕设内容不能作为一张平铺菜单直接生成。Skill 必须按以下四层理解和组合：

1. **业务项目大类**：回答“系统主要解决什么业务问题”，例如预约、报修、电商或课程学习。
2. **使用端形态**：回答“用户从哪里使用系统”，例如管理端网页、小程序或 APP。
3. **可复用能力**：回答“项目需要哪些通用模块”，例如用户、审核、评论、上传和统计。
4. **后台模块类型**：回答“每个模块按什么方式实现”，对应基础资料、关联资料、状态流程、库存核销、统计分析和系统配置。

项目母版负责把前三层组合成候选方案，真正开发时再把每项能力映射为第四层的具体后台模块。这样既能快速规划常见毕设，又不会绕过当前平台模板和真实代码。

### 24.1 能力状态标记

Skill 输出项目蓝图时，每项能力必须标记为以下一种状态：

| 标记 | 含义 |
|---|---|
| 候选 | 来自常见题型或母版建议，尚未由用户选择 |
| 已确认 | 用户已经纳入范围，但还未核对或实现 |
| 平台已有 | 在当前平台或目标项目中找到直接代码依据 |
| 待开发 | 已确认需要，但当前项目没有对应实现 |
| 已验证 | 已完成实现，并通过约定范围的检查或测试 |

禁止把“候选”“已确认”写成“平台已有”，也禁止仅凭目录名将能力标记为“已验证”。

### 24.2 项目蓝图形成顺序

1. 识别题目主业务大类和主闭环。
2. 选择管理端网页、小程序或 APP 等使用端形态。
3. 从母版中提取必选能力和可选能力。
4. 删除与题目无关、用户禁止或项目条件不支持的能力。
5. 对每项保留能力标记当前状态和实现难度。
6. 形成模块依赖、实施顺序、风险和验收清单。
7. 经用户确认后，再逐个模块进入现有七阶段工作流。

## 二十五、常见毕设业务项目大类

下表中的流程是常见方案参考，不代表每个题目必须全部采用。任何审核、评价、收藏、支付或推荐能力，都必须在内容确认阶段单独确认。

| 业务大类 | 细分方向与题目参考 | 常见主流程 | 典型模块组合 | 关键边界 |
|---|---|---|---|---|
| 后台信息管理类 | 宿舍、图书、仓库、资产、药品、合同、驾校管理；“基于 SpringBoot 的 XX 管理系统设计与实现” | 登录 → 角色权限 → 信息录入 → 查询筛选 → 修改删除 → 统计展示 | 用户与权限、基础资料、分类字典、查询统计 | 管理员后台为主；普通用户端和统计均按题目选择 |
| 预约、报名、申请类 | 体育馆、实验室、挂号、门票、家政、活动报名 | 用户选择对象并提交 → 管理员审核或确认 → 结果反馈 → 用户查看 | 可预约对象、申请记录、审核处理、结果查询 | 是否需要时间段、名额、审核、取消必须逐项确认 |
| 报修、工单、售后类 | 宿舍报修、物业报修、家电售后、设备维修、客服工单 | 用户提交问题 → 管理员受理 → 分配处理 → 更新进度 → 完成反馈 | 问题记录、工单处理、人员分配、处理记录、评价 | 分配人员、附件、评价和退回不是默认必选功能 |
| 电商销售类 | 图书、农产品、零食、花店、眼镜、课程销售 | 商品浏览 → 购物车 → 下单 → 模拟支付 → 订单处理 → 评论收藏 | 商品分类、商品、购物车、地址、订单、互动 | 模拟支付不连接真实资金渠道；退款、物流须另行确认 |
| 校园生活类 | 二手交易、社团、快递代取、失物招领、教材征订、体育活动 | 学生登录 → 发布、申请或报名 → 管理员处理 → 信息展示 → 状态反馈 | 用户、信息发布、申请审核、公告、互动 | 不同题目的交易、认领、报名流程不能混成一个通用状态 |
| 教育课程类 | 在线课程、图书借阅、绘本推荐、题库、编程学习 | 注册登录 → 浏览内容 → 收藏、购买、借阅或学习 → 评论 → 后台管理 | 课程或图书、分类、学习或借阅记录、收藏评论 | 购买、借阅和学习是不同闭环，必须选定主流程 |
| 社区生活服务类 | 志愿者、家政、宠物护理、养老、工具租赁、小区物业 | 用户发布需求 → 服务申请或预约 → 管理员处理 → 服务记录 → 评价反馈 | 服务项目、需求或预约、处理记录、评价公告 | 线下交接、库存、人员派单是否存在必须依据真实需求 |
| 文旅餐饮类 | 景区、民宿、酒店、火锅店、点餐、旅游推荐 | 用户浏览 → 预约或下单 → 管理员处理 → 用户评价 → 数据统计 | 展示内容、房型或菜品、预约订单、评论、统计 | 房态、桌位、库存、配送和支付均是独立复杂能力 |
| 运动娱乐类 | 健身房、足球青训、体育馆、马拉松、短剧、动漫 | 用户浏览 → 报名、预约或收藏 → 管理员管理 → 数据统计 | 活动或内容、报名预约、收藏、资讯、统计 | 兴趣场景不能代替业务规则，训练、比赛与推荐需分别设计 |
| 推荐系统类 | 图书、商品、美食、动漫、短剧、房源、旅游推荐 | 用户浏览、收藏或评分 → 形成推荐依据 → 生成结果 → 用户查看 | 内容资料、行为记录、推荐结果、热门排行 | 题目名称必须与实际推荐方法一致，不得用热门排行冒充算法 |
| 数据统计增强类 | 销售、用户增长、热门排行、订单、预约统计 | 业务数据产生 → 按口径汇总 → 图表展示 → 管理人员查看 | 指标接口、趋势、占比、排行、筛选条件 | 更适合作为增强模块；每个指标必须说明来源、范围和口径 |

### 25.1 大类选择原则

- 一个项目只确定一个主业务闭环，其他大类作为辅助能力来源。
- 题目包装可以新颖，但模块组合必须能落到真实数据和动作。
- 基础弱或周期短时优先选择后台信息管理、预约申请或报修工单。
- 电商、推荐、小程序和 APP 会引入更多联调与验收工作，不因“看起来高级”自动推荐。
- 论文题目、功能清单、页面展示和后台实现必须使用同一套业务边界。

### 25.2 题目命名规则

- 管理员后台为主时，可采用“基于 SpringBoot 的 XX 管理系统设计与实现”。
- 带普通用户服务闭环时，可采用“基于 SpringBoot 的 XX 服务平台设计与实现”。
- 小程序作为主要用户端时，可采用“基于微信小程序的 XX 服务平台设计与实现”。
- Android APP 作为主要用户端时，可采用“基于 Android 的 XX APP 设计与实现”。
- 只有实际完成并验证协同过滤时，才采用“基于协同过滤的 XX 推荐系统设计与实现”；否则应按真实的热门、标签或行为规则命名。
- “智能”“实时”“大数据”等词语不能只用于包装，必须对应可演示、可解释、可验收的实现。

## 二十六、使用端形态

小程序和 APP 是使用系统的入口，不是独立业务流程。它们必须与前述业务大类组合，例如“校园报修 + 小程序”或“农产品销售 + APP”。

| 使用端 | 适合场景 | 后台最低契约 | 额外验收 | 风险边界 |
|---|---|---|---|---|
| 管理端网页 | 所有后台管理类项目 | 登录身份、分页列表、详情、维护、权限 | 菜单按钮、查询条件、表单和接口对应 | 不默认需要普通用户前台 |
| 小程序前台 + Java 后台 | 跑腿、报修、点餐、失物招领、预约、二手交易 | 登录、列表、详情、提交、本人记录 | 小程序端请求、身份传递、图片访问、真机或开发工具联调 | 不默认具备微信支付、订阅消息、定位或扫码 |
| APP 前台 + Java 后台 | 电商、家政、评论、学习、宠物服务 | 登录、列表、详情、业务提交、订单或评论 | 安装环境、网络地址、权限、版本和设备联调 | 不默认具备推送、第三方登录、应用商店发布或真实支付 |

### 26.1 多端共用原则

- 管理端和用户端可以复用同一业务服务，但权限与数据范围必须分别检查。
- 小程序或 APP 接口只提供当前端实际需要的能力，不复制全部管理接口。
- 用户身份由后台登录机制确认，不能仅信任端侧提交的用户编号。
- 图片和附件必须验证上传、保存、访问和删除的完整链路。
- 只有完成对应端的真实联调，才能把该使用端标记为已验证。

## 二十七、可复用能力目录

| 能力模块 | 可复用功能 | 常见适用题型 | 最低验收与边界 |
|---|---|---|---|
| 用户模块 | 注册、登录、个人信息、修改密码 | 所有带普通用户端的项目 | 复用现有登录体系；注册是否开放需确认 |
| 管理员模块 | 用户、角色、权限、菜单管理 | 所有后台项目 | 优先复用平台系统管理，不另建重复体系 |
| 信息管理模块 | 新增、修改、删除、详情、分页、搜索、筛选 | 管理类及各类基础资料 | 查询条件必须真实生效，删除需检查关联影响 |
| 申请审核模块 | 提交申请、查看进度、审核处理、状态反馈 | 预约、报名、请假、报修、服务申请 | 是否允许撤回、拒绝、再次提交必须明确 |
| 订单模块 | 购物车、下单、订单状态、模拟支付 | 电商、课程销售、点餐 | 模拟支付仅改变演示状态，不处理真实资金 |
| 评论收藏模块 | 点赞、收藏、评论、评分 | 电商、课程、推荐、文旅、内容平台 | 防重复规则、删除权限和评分范围需要确认 |
| 公告资讯模块 | 公告发布、资讯管理、列表、详情 | 大多数项目 | 区分公告与业务数据，不默认增加审核发布流程 |
| 文件上传模块 | 商品图片、头像、问题附件、资料图片 | 商品、宠物、失物招领、报修等 | 文件类型、大小、存储位置和访问地址形成闭环 |
| 数据统计模块 | 柱状图、饼图、折线图、排行 | 各类项目增强 | 先定义指标口径，再选择图表，不返回无依据模拟数据 |
| 推荐模块 | 热门、分类、标签、浏览、收藏或评分推荐 | 图书、商品、美食、旅游、短剧等 | 明确规则或算法、输入依据、空数据和去重策略 |
| 小程序接口模块 | 登录、列表、详情、提交、本人记录 | 小程序项目 | 与小程序实际页面逐项对应，不默认微信生态能力 |
| APP 接口模块 | 登录、列表、详情、订单、评论 | APP 项目 | 与 APP 实际调用逐项对应，并完成设备侧联调 |

### 27.1 复用等级

- **平台复用**：当前公共框架已经提供，目标项目按规范接入。
- **模块复用**：当前项目中存在相近模块，可借鉴结构但不能复制业务规则。
- **方案复用**：只有大纲、状态或接口组合，需要重新映射当前题目。
- **不可复用**：涉及不同业务含义、数量规则或权限边界，必须重新设计。

## 二十八、项目母版组合

母版的作用是快速形成完整大纲，而不是一键复制代码。必选能力用于建立主闭环，可选增强必须经过确认。

| 母版 | 适合题型 | 必选能力 | 可选增强 | 主闭环与验收重点 |
|---|---|---|---|---|
| 通用后台管理母版 | 宿舍、仓库、资产、药品、合同、驾校 | 管理员、角色权限、基础资料、分页查询 | 文件、公告、统计 | 登录授权 → 资料维护 → 查询筛选；重点验收权限、字段和查询 |
| 预约报名母版 | 场馆、实验室、挂号、门票、家政、活动 | 用户、可申请对象、申请记录、处理结果 | 时间段、名额、取消、通知、统计 | 提交 → 校验 → 审核或确认 → 结果；重点验收重复、名额和状态 |
| 报修工单母版 | 宿舍、物业、设备、售后、客服 | 用户、问题提交、工单处理、进度记录 | 图片、人员分配、评价、统计 | 提交 → 受理 → 处理 → 完成；重点验收责任人、流转和历史记录 |
| 电商订单母版 | 图书、农产品、零食、花店、眼镜、点餐 | 商品分类、商品、购物车、地址、订单 | 模拟支付、收藏评论、销量统计 | 选购 → 下单 → 处理 → 完成；重点验收价格、数量和订单状态 |
| 校园服务母版 | 二手、社团、代取、失物、教材、体育活动 | 学生用户、信息发布、申请或报名、后台处理 | 公告、图片、评论、统计 | 发布或申请 → 处理 → 展示反馈；根据题目选定唯一主流程 |
| 课程图书母版 | 在线课程、图书借阅、绘本、题库、学习平台 | 内容分类、课程或图书、用户记录 | 收藏评论、购买、借阅、进度、推荐 | 先选择学习、购买或借阅之一作为主闭环，再配置对应记录和状态 |
| 小程序前台 + Java 后台母版 | 跑腿、报修、点餐、失物、家政、二手 | 选定业务母版、小程序接口、管理端处理 | 图片、位置、消息、统计 | 小程序发起 → 后台处理 → 用户查结果；只实现已确认的微信能力 |
| 推荐系统增强母版 | 图书、商品、美食、动漫、短剧、房源 | 内容资料、用户行为、推荐规则、结果接口 | 热门、标签、收藏、评分、排行 | 行为产生 → 规则计算 → 推荐展示；重点验收依据、可解释性和题目一致性 |

### 28.1 母版裁剪规则

- 先选主母版，再按题目增加一到两个增强能力，避免无边界堆模块。
- 母版中的必选能力仍需检查当前平台是否已有；没有则标记待开发。
- 可选增强没有得到用户确认时，不进入文件清单和验收范围。
- 组合两个母版时必须指定主闭环，另一个母版只能提供辅助能力。
- 每个后台模块仍需单独判断属于六类模块中的哪一类，并执行对应检查。

### 28.2 重点防夸大规则

- **模拟支付**：只允许描述为演示性的订单确认动作；没有真实渠道时不得写在线支付、退款、对账或资金结算。
- **推荐系统**：热门、分类、标签、浏览、收藏、评分和协同过滤必须区分；题目写协同过滤时必须存在可验证的实际计算过程。
- **数据统计**：图表不是统计依据；必须从真实业务数据按明确口径汇总，不能为展示效果返回固定数据。
- **审核流程**：不是所有预约和发布都需要审核；只有业务明确要求且后台存在处理入口时才加入。
- **评价反馈**：必须存在可评价对象、评价人资格和完成前提，不能只生成一张孤立评论表。
- **文件上传**：只有上传成功不算闭环，还要检查业务记录关联、访问展示和失效文件处理。
- **小程序与 APP**：只完成后台接口不等于客户端完成，必须分别报告后台验证和端侧联调状态。
- **题目包装**：论文标题中出现的算法、实时、智能或可视化能力，必须能在实现和验收中找到直接依据。

## 二十九、快速完成后台代码的最终目标

本 Skill 的最终目标不是只输出分析、建议或模块大纲，而是在需求边界明确后，快速生成并完善能够进入当前项目编译验证的后台代码。

### 29.1 正常交付结果

一次完整的后台模块任务应交付：

- 位于 `system.store.functionModule.<模块名>Module` 的业务模块代码。
- 与生成档位匹配的 Controller、Service、DAO、实体和 SQL。
- 业务需要时追加 DTO、VO、自定义 DAO 方法与 Mapper XML。
- 与真实接口一致的菜单和按钮权限 SQL。
- 字段、查询、权限、状态、事务和模板能力开关的实现结果。
- 编译、结构、数据库映射、权限和业务闭环检查报告。

### 29.2 “快速”的含义

- 已经从项目查明的公共结构不重复询问。
- 通过模块规格模板一次性收集会影响代码结果的信息。
- 基础骨架直接复用 KeyModule 与 ModuleMaker，不重新手写平行模板。
- 按生成档位只加载和生成当前模块需要的内容。
- 用户已经明确要求实施且规格完整时，该请求视为实施授权，不额外重复确认。
- 速度不能通过猜测字段、跳过权限、忽略 SQL 或省略验证获得。

### 29.3 不属于完成的结果

- 只生成实体或建表 SQL，没有 Controller 与 Service。
- 只复制 KeyModule，仍残留 `Key`、`key`、`XX` 或无关模板能力。
- 代码可以编译，但接口、权限 SQL、实体和数据表互不对应。
- 只输出代码片段，未写入用户要求的目标项目。
- 只报告“已完成”，没有列出实际文件和验证结果。

## 三十、模块规格契约

正式生成代码前，将用户需求和项目事实整理为一份模块规格。简单模块可以由 Skill 在对话中形成，复杂模块使用 `assets/module-spec-template.yaml` 保存并校验。

### 30.1 模块身份

| 内容 | 要求 |
|---|---|
| 中文名称 | 用于说明、接口文档、菜单和数据库注释 |
| 大驼峰名称 | 用于类名和 `<模块名>Module` 目录 |
| 小驼峰名称 | 用于变量、接口前缀和权限标识 |
| 下划线名称 | 用于数据表与 SQL 文件命名 |
| 目标包 | 默认 `system.store.functionModule.<模块名>Module` |
| 生成档位 | 快速资料型、标准关联型、业务流程型或统计查询型 |

### 30.2 数据字段

每个字段至少明确：业务名称、Java 类型、数据库类型、是否必填、默认值、长度或精度、是否唯一、是否查询、是否展示、是否导入导出、说明及关联对象。

以下字段不能只按模板默认处理：

- 金额、库存、名额、次数、评分等需要范围或精度的字段。
- 状态、类型等需要明确取值含义的字段。
- 用户、课程、商品等关联字段。
- 创建时间、更新时间、创建人和逻辑删除等公共字段。
- 富文本、图片和附件等需要存储与访问约定的字段。

### 30.3 查询与数据范围

- 后台分页需要哪些精确、模糊、时间范围和关联条件。
- 是否存在公开详情或公开分页。
- 管理员、普通用户、创建人、教练或其他角色分别能看到哪些记录。
- 列表是否返回关联名称、统计值或组合信息。
- 是否需要排序、排行、导入、导出或聚合统计。

### 30.4 写入与业务动作

- 基础新增、修改、逻辑删除和恢复是否保留。
- 是否存在提交、审核、发放、归还、结课、取消等专用动作。
- 每个动作的执行角色、前置状态、校验、数据变化和结果状态。
- 哪些动作需要同时修改多项数据并使用事务。
- 哪些动作需要记录操作者、时间、原因或业务日志。

### 30.5 模板能力开关

规格必须显式记录：公开查询、恢复、批量状态、批量排序、低代码元数据、Excel 导入、Excel 导出、权限 SQL是否启用。未填写的高风险能力按第三章默认值处理，不能因为模板中存在就自动开放。

### 30.6 规格确认状态

- **草稿**：仍有影响代码结果的缺失项，只能继续分析。
- **可生成**：命名、字段、权限、动作和数据库范围完整，可以预演文件。
- **已授权**：用户明确要求实施，允许写入项目。
- **已实现**：文件已生成或修改，但尚未完成全部验收。
- **已验收**：满足第三十四章完成标准。

### 30.7 从项目需求拆分模块规格

- 先按业务对象和闭环责任拆分模块，不按角色数量重复创建模块。
- 学员端、教练端和管理员端访问同一业务对象时，共用实体、DAO 与核心 Service，分别定义接口、权限和数据范围。
- 用户、角色、菜单、个人资料等平台公共能力优先映射到既有系统模块，不在 `functionModule` 下重复生成。
- 一条核心流程跨越多个模块时，明确主记录、关联记录和负责事务编排的 Service。
- 每个模块规格保存来源需求编号与流程编号，生成文件后回填实际类和接口位置。
- 最新需求删除某项功能时，根据追踪关系同时检查接口、Service、权限 SQL、菜单和管理端调用是否需要移除。

## 三十一、后台代码生成档位

### 31.1 快速资料型

适用于宿舍、分类、公告、资产、药品等以单表维护为主的模块。

- 必选：Controller、Service、DAO、实体、建表与权限 SQL。
- 按需：公开查询、恢复、批量状态、批量排序、元数据、导入导出。
- 默认不生成：DTO、VO、自定义 Mapper XML、复杂状态动作。

### 31.2 标准关联型

适用于报名、课程订单、收藏、借阅记录等存在用户或业务对象关联的模块。

- 包含快速资料型的必要文件。
- 按需增加创建或修改 DTO、组合展示 VO、关联查询和重复业务校验。
- MyBatis-Plus 能完成时继续使用 Service 查询；复杂联表再增加 Mapper XML。
- 必须定义本人数据、管理数据和关联对象不存在时的处理。

### 31.3 业务流程型

适用于审核、工单、订单、发放归还、报损和结课等存在状态流转的模块。

- 包含标准关联型需要的文件。
- 增加专用业务动作、状态校验、多项写入事务和操作记录。
- 禁止通用编辑接口直接改变关键状态、库存或最终结果。
- 必须覆盖重复操作、非法状态、失败回退和终态处理。

### 31.4 统计查询型

适用于趋势、占比、排行、增长和仪表盘接口。

- 以 Controller、Service、DAO 和返回结构为主。
- 只读统计不强制生成新增、修改、删除与独立数据表。
- 单表聚合可直接查询；复杂联表聚合按需增加 Mapper XML。
- 必须说明指标来源、时间范围、筛选条件、空数据和排序口径。

### 31.5 档位升级规则

- 出现关联名称、重复业务校验或本人数据范围时，从快速资料型升级为标准关联型。
- 出现状态动作、多表写入、库存或核销时，升级为业务流程型。
- 统计只是业务模块的附加接口时，不拆成独立模块；存在多指标仪表盘时再使用统计查询型。
- 档位只能决定生成深度，不能替代真实业务规格。

## 三十二、代码文件职责与生成规则

| 文件 | 必须生成的内容 | 禁止做法 |
|---|---|---|
| Controller | 接口路径、请求接收、权限、统一响应和接口说明 | 在入口中堆积状态流转、库存扣减等核心业务 |
| Service | 查询、数据范围、业务校验、事务、状态变化和操作记录 | 只调用保存方法却宣称完整业务闭环 |
| DAO | 继承平台基础数据访问能力，按需声明自定义查询 | 为简单单表操作重复编写无意义方法 |
| Entity | 数据表映射、真实持久化字段、逻辑删除和必要说明 | 混入只用于展示的关联名称或统计结果 |
| SQL | 新建表或变更脚本、字段注释、索引约束、菜单按钮权限 | 未确认时执行脚本，或用删表重建修改已有表 |
| DTO | 接收创建、修改、结算等专用输入 | 直接信任用户编号、状态和计算结果等敏感输入 |
| VO | 返回关联名称、组合详情或统计展示结果 | 把 VO 字段误写入实体和数据表 |
| Mapper XML | 实现确有必要的复杂联表或聚合查询 | 与 DAO 方法、参数或返回类型不一致 |

### 32.1 Controller 生成规则

- 路径前缀、类名、Service 引用和权限标识统一来自模块规格。
- 只保留模板能力开关中启用的接口。
- 公开接口必须由规格明确允许，后台管理接口使用现有权限结构。
- 关键状态变化使用专用接口，不通过通用修改入口完成。
- 输入结构复杂或包含敏感字段时使用专用 DTO。

### 32.2 Service 生成规则

- 沿用当前平台的具体 Service 类与 `ServiceImpl<DAO, Entity>` 结构，不额外生成无依据的 Service 接口层。
- 统一处理数据范围、关联存在性、重复业务、状态和数量校验。
- 多表或多数量变化使用事务，任何失败不得留下半完成数据。
- 只实现规格中真实存在的筛选和排序，不保留无法生效的示例条件。
- 业务日志说明动作和结果，不记录密码等敏感内容。

### 32.3 DAO 与 Mapper 生成规则

- 基础 DAO 继承 `BaseMapper<Entity>` 并保持平台注解。
- 简单条件查询由 Service 与 MyBatis-Plus 完成。
- 自定义 SQL 方法只有在真实查询需要时增加，并同步生成或修改 Mapper XML。
- Mapper XML 默认放在 `src/main/resources/mapper/function`，命名空间对应完整 DAO 包路径。

### 32.4 Entity 生成规则

- 表名、主键、逻辑删除和字段名称必须与 SQL 一致。
- 保留当前低代码平台需要的实体标记和字段说明。
- Excel 注解仅服务于确认启用的导入导出能力。
- 展示型关联名称、统计值和临时计算结果放入 VO，不污染实体。

### 32.5 SQL 生成规则

- 新模块生成新建表脚本；已有模块变更生成增量修改方案，不默认删表重建。
- KeyModule 中的删表语句只可作为新表初始化模板内容，Skill 不自动执行数据库脚本。
- 根据真实查询和唯一性规则增加必要索引或约束，不机械为每个字段建索引。
- 菜单、页面和按钮权限只注册实际保留的能力，并与 Controller 权限完全一致。
- SQL 文件生成完成不等于数据库已经执行，交付报告必须区分“已生成”和“已执行验证”。

## 三十三、快速代码生成流程

### 步骤一：接收项目需求

- 读取用户提供的按角色功能清单、核心流程、最新修正和禁止事项。
- 为每项功能和每条流程分配追踪编号，不改变用户原始业务含义。
- 项目母版只用于发现可能缺失的说明，不自动加入开发范围。

### 步骤二：快速扫描

- 定位平台源码根、KeyModule、ModuleMaker、目标模块和管理端调用。
- 若本次任务与刚完成的同一项目扫描结果一致，可复用结果，但写入前仍检查目标目录和工作区变化。

### 步骤三：需求与实现对照

- 生成角色功能覆盖矩阵和核心流程追踪矩阵。
- 将每项能力标记为平台已有、部分实现、待开发、待移除或已验证。
- 找出功能与流程不一致、角色冲突、孤立操作和缺少异常去向的问题。

### 步骤四：形成模块规格

- 从用户描述提取模块身份、字段、角色、查询、动作和能力开关。
- 能从项目查明的内容自动填写；影响业务结果的缺失项集中列出。
- 简单模块规格完整且用户已要求实施时，直接进入预演，不重复询问。

### 步骤五：规格校验与文件预演

- 选择生成档位并校验模块规格。
- 列出将创建、修改、保留和跳过的文件。
- 检查目标目录、同名类、数据表、权限标识和工作区冲突。

### 步骤六：生成基础骨架

- 新模块优先通过 KeyModule 与 ModuleMaker 生成五类基础文件。
- 目标模块已存在时禁止重新生成，转为增量增强。
- 立即检查并清除残留占位符，按能力开关裁剪模板入口。

### 步骤七：完成业务增强

- 按档位增加字段、查询、DTO、VO、专用动作、事务或 Mapper XML。
- 同步修改 Controller、Service、DAO、实体、SQL、权限和管理端契约。
- 不修改与规格无关的公共框架和其他模块。

### 步骤八：自动验收

- 检查目录包路径、命名、占位符、实体与 SQL、接口与权限、DAO 与 Mapper。
- 编译后台，并按模块类型执行必要的正常、重复、越权、非法状态和数量测试。
- 前端在任务范围内时，继续验证管理端服务调用与构建。

### 步骤九：交付报告

- 列出实际创建和修改的文件。
- 列出保留与关闭的模板能力。
- 报告通过、警告、阻断和未验证项目。
- 明确 SQL 是否只生成、是否执行，以及是否还需要用户提供运行环境。
- 回填功能覆盖矩阵与流程追踪矩阵，确保每项交付都有需求来源。
