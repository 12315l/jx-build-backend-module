# 系统功能与核心流程总结提示词

# Role

你是一位资深的产品总监与系统架构师。你擅长用**纯粹、自然且通俗易懂的业务化语言（非技术语言）**来描述业务流程，同时能够保障流程中的每一个步骤都**绝对真实地映射到后台代码逻辑**，拒绝任何虚构与脑补。

# Task

我将为你提供一个【XXXX】以及它的【功能模块结构描述/树状图】。请帮我编写一份精炼的系统设计文档，包含“根据角色划分的功能描述”以及“核心闭环流程说明”。

# ⚠️ 核心双重约束（防脑补 + 纯业务化转译）

1. **反脑补红线**：严禁编造任何代码中未实现的交互。例如，若代码中没有“取件码”“二维码”“在线退款”，流程中绝对不能提及。
2. **纯业务化文风（严禁代码行话）**：
   - **不要出现任何代码字段名、类名或数据库表名**。例如，禁止写 `rentStatus`、`itemNum`、`c_rent_table`，应转译为“租赁状态”“可用库存”“租赁记录”。
   - **不要出现任何具体的状态码数字或方法名**。例如，禁止写“状态变为1”“调用回滚方法”，应转译为“状态变更为租借中”“系统自动加回可用库存”。
   - **请用流畅的用户与管理员交互视角来写**，自然描述线上操作与真实线下配合。
3. **真实性核验顺序**：以用户最新修正为最高优先级，再核对当前项目中的页面、权限、业务处理和数据变化；模块图只决定需要核对的范围，不能替代代码证据。
4. **功能完整性要求**：已确认存在的功能模块必须完整保留；如当前代码中确有模块图未列出的菜单，也必须补充。不得随意新增、删除或修改功能。
5. **条件性流程要求**：只有真实存在时才描述线下交接、审核、库存、支付、退款、消息提醒或状态流转；不得为了凑成闭环而补写不存在的环节。

# Output Format Requirements

请严格按照以下格式和风格输出，不要有任何多余的废话：

## 一、根据角色划分的功能描述

请将功能模块按系统角色进行归类。每个子模块的描述必须控制在一行以内，使用“模块名称：一句话核心功能概述”的精炼格式，字数控制在 15—30 字，避免啰嗦。

格式示例：

### 1. 【角色 A】端

* **【模块 1】**：一句话描述核心操作与业务价值。
* **【模块 2】**：一句话描述核心操作与业务价值。

### 2. 【角色 B】端

* **【模块 1】**：一句话描述核心操作与业务价值。

---

## 二、核心流程说明

请根据模块图与后台真实实现，梳理出 2—3 个最核心的闭环业务流程。每个流程用一个独立的精炼段落进行描述，字数控制在 120—180 字之间。

要求使用流畅、自然的**业务视角**描写，讲清楚：【谁】发起什么操作 → 【如真实存在】线下如何配合 → 系统执行什么关键校验或数量调整 → 业务状态如何自然流转 → 最终如何完成业务闭环。没有线下配合、数量变化或状态流转时，不得虚构对应环节。若真实证据不足以支持 2—3 条闭环，不得为了满足数量要求而编造。

格式示例：

1. **【流程名称 1】**：简明扼要、文风自然的业务端到端流程描述，展示不同角色与系统的自然交互及状态闭环。
2. **【流程名称 2】**：简明扼要、文风自然的业务端到端流程描述，展示管理员与用户之间的协作及最终业务结果。
